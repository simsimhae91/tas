- **Scaffolding-only Wave 0** — Plan 01 deliberately avoids any behavioral code change. All 3 stubs exit 0, REQUIREMENTS.md §C.1 entries are [ ] (Pending), canaries.md Canary #7 is PENDING. This keeps Wave 0 diff small (pure additive) and lets Plans 02-07 own their respective behavior changes under dedicated tasks.
- **Bash stub delegates to Python stub (not duplicate logic)** — `simulate_subagent_orphan.sh` uses `exec python3 "$PYTHON_HARNESS" "$@"` so there is ONE source of truth for the PENDING message. Plan 06 will keep this pattern: the bash wrapper remains a thin delegate; the Python harness owns the orphan-survival assertions.
- **Canary #7 placeholder front-loads the planned Pass criteria** — the PENDING section includes the exact Pass criteria (subagent duration < 10s, PPID=1, marker=="survived", wall-clock bounds) verbatim from D-VERIFY-TOPO-01. Plan 06 Task 2 becomes a single-section text swap (strip "(planned)" / "(Wave 0 stub)" qualifiers, flip Status from PENDING to active) rather than authoring fresh prose.
- **Strict-mode bash for test_exit_trap.sh** — chose `set -euo pipefail` (Canary #6 uses only `set -u`) because Plan 02's full implementation will run 4 sequential test cases and needs failure propagation. Stub inherits the strict mode so Plan 02 doesn't have to change the shebang/mode line.
