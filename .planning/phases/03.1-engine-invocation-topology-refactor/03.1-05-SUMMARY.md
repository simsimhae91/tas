**1. [Rule 3 - Blocking] Adjusted plan's awk range pattern for bullet-in-block acceptance grep**

- **Found during:** Task 1 verification
- **Issue:** Plan's acceptance check `test "$(awk '/<!-- Phase 0b does NOT/,/-->/' skills/tas/SKILL.md | grep -c 'Introduce new halt_reason enums')" -eq 1` expected a single line matching `<!-- Phase 0b does NOT`, but SKILL.md has `<!--` on L300 and `Phase 0b does NOT:` on L301 (line-break between the comment opener and the label). awk's range-pattern `/<!-- Phase 0b does NOT/,/-->/` requires a single line matching both tokens, so it produced empty output and the grep counted 0 even when the bullet was correctly placed.
- **Fix:** Executor ran the semantically-equivalent adjusted pattern `awk '/<!--$/,/-->/' skills/tas/SKILL.md | grep -c 'Introduce new halt_reason enums'` which matches from the `<!--` line (first HTML comment opener in Phase 0b) through the closing `-->`. Result: `1` (correct). The semantic invariant (bullet placed inside the Phase 0b prohibition HTML comment block) is fully verified; the plan's grep syntax just did not fit the file's line structure.
- **Files modified:** none additional — this was a verification-command tuning, not a file-edit deviation. SKILL.md edit itself passed the plan's intent verbatim.
- **Commit:** n/a (executor-side tuning, not a file change)
- **Pattern:** Same phase-level maturity pattern as Plans 03 (1x Rule 3 auto-fix) and 04 (5x). Phase 3.1 total = 7 Rule 3 plan-verbatim-vs-grep auto-fixes across Plans 02-05.
