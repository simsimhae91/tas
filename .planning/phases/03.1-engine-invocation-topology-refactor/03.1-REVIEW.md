# Phase 3.1: Code Review Report

**Reviewed:** 2026-04-21T16:25:00Z
**Depth:** standard

## Summary

Phase 3.1 (engine-invocation topology refactor) implements Scenario B: MetaAgent
now spawns `run-dialectic.sh` as a fire-and-forget orphan via `nohup ... &
echo $!` and polls `engine.done` / `engine.exit` markers locally. The refactor
eliminates the Claude Code harness `bash_id` reap hazard that caused the Phase 3
close blocker. A defensive EXIT trap in `run-dialectic.sh` (TOPO-01) and two
new regression tests (`test_exit_trap.sh`, `simulate_subagent_orphan.py`) guard
the critical invariant that `exec` must not appear on the final invocation
lines. Documentation (CLAUDE.md, engine-invocation-protocol.md, meta.md,
canaries.md) is unusually thorough and self-consistent — most cross-file
references line up byte-for-byte.

The refactor is mechanically correct and the test coverage is strong. The
issues below are modest: one latent edge-case in the EXIT trap when
`run-dialectic.sh` is invoked with no args, one gap in the Phase 2 integration
test (it exercises the trap but not the `nohup + &` spawn chain in
combination), and several documentation / style items.

No critical issues — no security vulnerabilities, no correctness bugs on the
happy path, no information-hiding regressions, and the `grep -c "^[[:space:]]*exec "`
static invariant currently returns `0` as required.

## Warnings

### WR-01: EXIT trap silently degrades when `run-dialectic.sh` is invoked with no arguments

**File:** `skills/tas/runtime/run-dialectic.sh:45-46`

**Issue:** The trap derives `LOG_DIR` from `$1`:

```bash
LOG_DIR="$(dirname "$1")"
trap 'rc=$?; echo "$rc" > "${LOG_DIR}/engine.exit.tmp" && mv -f ... ' EXIT
```

If the script is invoked with zero arguments (`bash run-dialectic.sh`), `$1`
expands to the empty string and `dirname ""` returns `.`. The trap then writes
`engine.done` / `engine.exit` into the caller's current working directory —
e.g. `$HOME` if a user invokes the script directly, or wherever `/tas` happened
to be running from.

Empirically verified: `bash -c 'LOG_DIR="$(dirname "")"; echo "[$LOG_DIR]"'`
prints `[.]`. The EXIT trap then fires with `rc=...; echo "$rc" > "./engine.exit.tmp"`,
creating stray files outside any workspace.

MetaAgent always passes a step-config.json path (meta.md Phase 2d step 7), so
the production path is safe. But: (a) the script is publicly callable, (b) the
failure mode is silent marker pollution of cwd, and (c) an early-abort edge
case (script run accidentally during a copy/paste) would now leak files.

**Fix:** Add a preflight check ahead of the trap:

```bash
if [ $# -lt 1 ] || [ -z "$1" ]; then
  echo "ERROR: run-dialectic.sh requires a step-config.json path as \$1" >&2
  exit 2
fi
LOG_DIR="$(dirname "$1")"
if [ ! -d "$LOG_DIR" ]; then
  echo "ERROR: LOG_DIR derived from \$1 does not exist: $LOG_DIR" >&2
  exit 2
fi
trap '...' EXIT
```

The preflight runs BEFORE the trap is installed, so a bail exits cleanly
without attempting marker writes. (Alternatively: install the trap only after
confirming `$1` is a real file path — but the explicit preflight is more
legible and matches the existing `find_python` bail-on-missing pattern at
lines 28–33.)

### WR-02: Canary #7 Phase 2 does not exercise the `nohup ... & echo $!` spawn chain

**File:** `skills/tas/runtime/tests/simulate_subagent_orphan.py:236-248`
**File:** `skills/tas/runtime/tests/test_exit_trap.sh:155`

**Issue:** Phase 2 (`_phase_2_real_chain_integration`) invokes the modified
`run-dialectic.sh` directly via `subprocess.run(["bash", str(mod_script), str(step_cfg)])`
with captured stdout. The mirrored Test E in `test_exit_trap.sh` does
`bash "$MOD_SCRIPT" "$E_STEP_CFG" > "${E_DIR}/engine.log" 2>&1`.

Both invocations are foreground, blocking, and stdout-captured — the
production invocation is:

```
nohup bash run-dialectic.sh <cfg> > engine.log 2>&1 & echo $!
```

Phase 2 thus verifies the EXIT trap fires under a *simpler* invocation than
production. This is sufficient for Issue #1 (catching `exec` reintroduction,
which would break the trap under any invocation), but it leaves a coverage
gap: a hypothetical future regression that interacts specifically with
`nohup ... &` backgrounding (e.g. SIGHUP handling inside the trap body, or
stdout-redirect timing racing the trap's `echo "$rc" > ...`) would not be
caught by Phase 2. Phase 1 exercises `nohup + &` but with a synthetic payload,
not the real `run-dialectic.sh`.

**Fix (choose one):**

Option A — add a Phase 3 sub-assertion that combines both: `nohup bash
run-dialectic-mod.sh ... & PID=$!; wait $PID`, then check markers. This
verifies the real chain under the real spawn pattern.

Option B — document explicitly in `canaries.md` §Canary #7 and
`simulate_subagent_orphan.py` docstring that Phase 2 is intentionally
foreground and that the `nohup`+`&` invocation is covered only by Phase 1's
synthetic payload. The current docstring mentions "real chain integration"
which is technically true but elides the spawn-chain coverage gap.

Option A is worth ~20 lines of Python and closes the gap. Option B is a
documentation-only fix but preserves the current signal. Either is acceptable;
Option A is the stronger guard if the test budget allows.

## Info

### IN-01: Step numbering skips from 8 to 8b to 9 in meta.md Phase 2d

**File:** `skills/tas/agents/meta.md:494-647`

**Issue:** The step loop numbering reads:

```
7. Spawn engine
8. Poll engine completion locally
8b. Classify verdict
9. Read deliverable
9.5. Update checkpoint.json
```

`8b` is unusual — the conventional pattern would be a fresh ordinal (9),
shifting downstream to 10 / 10.5. The current scheme works but readers
scanning for "step 9" may find "Read deliverable" before realizing "Classify
verdict" is logically between polling and reading.

**Fix:** Cosmetic. Either renumber 8b → 9 (shifting 9 → 10 and 9.5 → 10.5),
or add an explicit cross-reference note at the top of 8b:
`8b. (inserted between 8 and 9 — classify polling result before reading deliverable)`.
Renumbering is riskier because `step 8` is referenced from the Edge Cases
table (meta.md:786) and from `engine-invocation-protocol.md` (line 127, 150):
these would all need synchronized updates. The cross-reference note is the
safer fix.

### IN-02: `simulate_subagent_orphan.py` `timeout=30` is dead-code margin for Phase 2

**File:** `skills/tas/runtime/tests/simulate_subagent_orphan.py:241`

**Issue:** The Phase 2 invocation sets `timeout=30` on the `subprocess.run`
call. The mock dialectic script exits immediately (`exit 0` after
`echo '{...}'`), and `run-dialectic.sh`'s `WATCHDOG_TIMEOUT_SEC` default is
1200s — but since the mock terminates in ms, the watchdog never fires. The
30s timeout will only trip if the trap itself hangs (which would be a genuine
regression worth catching).

This is defensive and correct — just worth a one-line comment explaining why
30s was chosen over, say, 5s (which would be tight for CI hosts under load).

**Fix:** Add a comment on line 241:

```python
            # 30s is generous — mock exits in ms; timeout trips only if the
            # EXIT trap body hangs, which is a worthwhile regression to catch.
            timeout=30,
```

### IN-03: `canaries.md` Canary #7 docstring says "~135s wall-clock including Phase 2" but Phase 2 runs the mock directly (no DURATION)

**File:** `skills/tas-verify/canaries.md:241`

**Issue:** The comment `Default CI mode (T=120s; ~135s wall-clock including Phase 2)`

### IN-04: `_pid_ppid` returns -1 on both "PID not found" and "ps output unparseable", making failure diagnosis harder

**File:** `skills/tas/runtime/tests/simulate_subagent_orphan.py:53-63`

**Issue:** The function catches both `subprocess.CalledProcessError` (ps
exited non-zero — PID does not exist) and `ValueError` (int() failed on ps
output), returning `-1` for both. Downstream, the caller (`_phase_1_orphan_survival`
line 138) prints `FAIL: Phase 1 PPID={ppid} for PID {pid}, expected 1` —
which on ValueError would print `PPID=-1`, an ambiguous signal.

Two distinct failure modes collapse to one symptom: (a) macOS launchd
reparenting to a non-1 PID (legitimate signal), (b) PID genuinely died
between the `_pid_alive` check and the `_pid_ppid` call (race — should retry
or flag as `died_during_ppid_check`).

**Fix:** Distinguish the two cases:

```python
def _pid_ppid(pid: int) -> int:
    try:
        out = subprocess.check_output(
            ["ps", "-o", "ppid=", "-p", str(pid)],
            stderr=subprocess.DEVNULL,
        ).decode().strip()
    except subprocess.CalledProcessError:
        return -1  # PID not found (race: died between _pid_alive and _pid_ppid)
    try:
        return int(out) if out else -2  # ps succeeded but output empty (macOS edge case)
    except ValueError:
        return -2  # ps output unparseable
```

Then distinguish `-1` vs `-2` in the caller's FAIL message. Low-priority —
the current behavior is correct in the common case; this is diagnostics polish.

### IN-05: Comment drift in `run-dialectic.sh`:46 — "tempfile + mv -f" is slightly inaccurate

**File:** `skills/tas/runtime/run-dialectic.sh:40-41`

**Issue:** The comment claims "atomic via tempfile + mv -f (POSIX rename())".
The code uses a `.tmp` suffix inside the SAME directory (`${LOG_DIR}/engine.exit.tmp`),
not a true tempfile (e.g. `mktemp`). This is idiomatic and correct for POSIX
`rename()` atomicity (source and destination must be on the same filesystem,
which the same-directory convention guarantees), but "tempfile" is misleading
— `mktemp` is not involved.

**Fix:** Tighten the comment:

```
# atomic via in-directory .tmp + mv -f (POSIX rename() on same filesystem),
# matching Phase 1 checkpoint.py _atomic_write_json.
```

This is pure wording; no behavioral change.
